Simulator Tips

iOS Simulator Tips

  • In XCode 6+ disable "Slow Animations" to speed up testing. This option can be found in the Simulator Debug menu.
  • In XCode 7.X disable "Optimize Rendering for Window Scale" to prevent corrupt screen shots. This option can be found in the Simulator Debug menu.

ADB Simulator Tips

  • Don't close the simulator between sessions. If you leave the simulator booted and start the framework with it's device name, Soda will attach to the already booted device—saving time.
  • You can change the device's orientation by pressing Ctrl+F12

Soda VisualEditor UI

Menubar Buttons

  • The "Refresh DOM" button syncs the screen shot and DOM tree will the current screen or page on the running device (simulator or web page).
  • The "Download Current Screenshot" button downloads the current DOM screenshot.
  • The "Download Current DOM" button downloads the DOM tree in its raw JSON format.
  • The "Download Current Script" button downloads the DOM tree in its raw JSON format.
  • The "Shutdown Framework" button stops the running framework (Instruments, ADB, Selenium, etc) and close the editor window. You will be redirected to the Launch page to start another framework, if desired.
  • The "Shutdown Server" button kills the running Soda instance. This mean, the Soda VisualEditor will no longer be available and the server will shutdown.

Working with the screen

  • Hover over an element to see its bounds. If you switch to the Element Inspector toolbox, the element's attributes will be displayed when hovered over.
  • To see outlines of each element's position on the screen, you can toggle "Outline Mode" by pressing Ctrl+O.
  • Press the button in the upper-right hand corner of the page to refresh the screen image and element tree.
  • To prevent the Element Inspector from changing values accidentally when moving your cursor, you can "sticky" or "pin" the current element by clicking it. To "un-pin" the element, click it again.
  • You can remove (or stash) an element from the screen by right clicking it. This is useful is you'd like to view the attributes of an element that is underneith another. To add all the stashed elements back to the screen press Ctrl+R.
  • Different colors represent different element types.
  • Drag actions from the Actions toolbox on top of screen elements to write tests.

VisualEditor Toolboxes (Left)

Test Runner

  • The Test Runner allows you to run any test, suite, or module from the respective sub-menus.
  • To run an individual Test, click on the Test tab, and click on the desired Test.
  • To run an entire Module, click on the Module tab, and click on the desired Module.
  • To run an entire Suite, click on the Suite tab, and click on the desired Suite.
  • Once a suite, module, or test has been started, you will be switched to the Test Monitor toolbox automatically. From here you can control (stop, pause, etc.) the current test and view test output.
  • Once a test, suite, module run has been started, this tab will be "locked out" until the currently running item has finished.

Test Monitor

  • The Test Monitor allows you to control a running test, suite, or module.
  • This tab will remain "locked out" until a run has been started from the Test Runner toolbox.
  • The "Test Info" heading will update to show you important information about the current run as it becomes available.
  • The "Test Information" heading shows you the meta-data in regard to the running test (i.e. The test's name, id, module, suite, and description).
  • If a suite or module was run, the information shown will update as the suite or module moves to the next test.
  • The "Current Action" heading shows the action that is currently being executed against the running device.
  • The "Last Action" heading shows the action that was last executed against the running device.
  • The "Test Output" heading shows the Soda process output that is pertinent to testing only. Green output indicates that an action has passed, red indicates that it has failed.
  • To stop a test (or suite, or module) press the button. The stop button may not always be available.
  • To pause a test press the button. The pause button may not always be available.
  • While a test is paused, the "last" , "next" , and "stop" options will become available.
  • Pressing the "last" button will repeat the previous action.
  • Pressing the "next" button will execute the next action.
  • If a test fails, the "last" , "skip" and "continue" options will become available.
  • Pressing the "continue" button will fail the test, and the results will be displayed in the Result History toolbox.
  • Pressing the "skip" button will skip the failed test item, and continue onto the next action.
  • If a test passes, the results will be displayed in the Result History toolbox.

Result History

  • The Result History Toolbox shows the history for test, suite, and module runs duing this Soda VisualEditor session.
  • To view history for runs prior to this session, see the JSON files in the "test-results" directory listed in your project folder.
  • To clear the history list, click the "Clear" button.

Project Manager

  • The Project Manager Toolbox provides a tree-view of your Soda project folder.
  • Suites reside a the top level, followed by modules. Modules contain a folder for each action type. Within these folders are the respective action files for each action type.
  • To create a new suite, press the " Add New Suite" button. A modal will be displayed that will ask you to enter the filename of the suite. The suite's "suite.json" file will be automatically generated.
  • To create a new module, press the button on the suite where you wish to create the module. A modal will be displayed that will ask you to enter the filename of the module. The "module.json" and all action type subfolders will automatically be created for you.
  • To "load" an action into the Test Editor for editing, click the button to the right of the action name. You will be switched to the test editor toolbox automatically.
  • To delete an action, click the button to the right of the action name.
  • To run an action, press the button to the right of the action name. This will execute the aciton in "non-interactive" mode. So, running tests from here won't allow you to pause or stop them. Once an action is started, it will run through to completion or stop on a failure. The results will be displayed in the context menu that will appear at the end of the run.

Screen Elements (DOM Viewer)

  • The Screen Elements Toolbox lists all elements currently within the DOM (screen or page).
  • Elements are listed by hierarchy, with root level elements at the top of the DOM tree, and children elements listed beneith their parents.
  • To expand an element and see its children, click on the element name or the icon.
  • If an element has the icon, it indicates that it has no children (a leaf node on the DOM tree).
  • When you hover over a DOM element, it will be highlighted on the screen viewer.
  • To "pin" the element to the screen and prevent hovering from changing the displayed element, click the icon, on the right side of the element listing.
  • The "Query Selector" input field allows you to see which elements will match the entered selector. To test elements against a selector, enter the selector into the input field. Elements that match will display a purple icon.

Test Editor

  • The Test Editor allows you to dynamically create new tests without having to write any JSON.
  • To add test items to the list, drag them from the Actions Toolbox , onto the "Test Actions" (or "Screen Actions", or "Menu Actions", etc.) panel.
  • To clear a list, click the "Menu" button and select "Clear List". A dialog will appear asking for confirmation.
  • To download a test's JSON content, click the "Menu" button and select "Download List".
  • To save a test (screen, menu, or popup), click the "Menu" button and select "Save List". A dialog will appear asking for a filename to save the test.
  • Once you've added some actions, and you're ready to save the test (or screen, menu, or popup), select a Suite and Module from above the Actions panel to save the test in. Then fill out the test meta-data beneith the "Test Metadata" heading.
  • The "Test Name" field should relate directly to the test filename. While this isn't strictly enforced, if you save the test with a different value than what is listed in the "Test Name" field, it will be overwritten by the actual filename the next time the test is loaded into the test editor.
  • If you load a test into the test editor, then save it with a new filename or change the suite (and/or module), it will be save as a new test and the old test will be unaltered. Basically, the editor treats this like "Save As" in most text editors.

VisualEditor Toolboxes (Right)

Attributes Inspector

  • The Attributes Inspector shows you each element's respective properties.
  • Nested elements represent child elements of an object. For example, The "x" property is a child of the "origin" object—which is a child of the "rect" object.
  • The collection of these properties make up a screen element. All screen elements combined make up the current DOM.
  • The attributes inspector is useful for deciding on selectors when creating tests, and can be used for debugging purposes.

Actions

  • The Actions Toolbox lists all available testing actions that may be performed for each action file type (i.e. "test", "action", "screen", etc.).
  • To view the list of actions for a respective action file type, click the type's name on the button toolbar at the top of the screen.
  • To change the current syntax (a set of actions repsective to the running framework), click the dropdown at the top of the toolbox and select the desired syntax. The actions list will be automatically updated.
  • To write tests, drag an action onto either a screen element, or the Test Editor toolbox panel on the left side of the page.
  • Actions may only be dragged into the Test Editor panel if the action is the same as the selected test type.

Console

  • The console toolbox allows you to view terminal output from the currently running Soda process.
  • Any command that can be executed in the Soda REPL can be executed by entering the command in the Console input field and clicking "execute".
  • The console also acts as a JavaScript interpreter. Any JavaScript entered will be evaluated accordingly.
  • To clear the console, click the "Clear" button.

Settings

  • The settings toolbox shows all currently set Soda configuration values.
  • You can change a value, by clicking on a value and editing the value's text.
  • Be warned, that changing configuration values may break the running Soda process, or cause other unintended harm.

Keyboard Shortcuts

UI Shortcuts

  • Note, on windows the following would use the "Windows" key and not the "Control" key!
  • Ctrl+Space: Refresh the DOM
  • Ctrl+R: Add stashed elements back to the screen
  • Ctrl+O: Toggle Outline Mode
  • Ctrl+I: Download Screenshot
  • Ctrl+D: Download DOM Tree

Writing Tests

  • Ctrl+G: Generate Screen for current action list (if available)
  • Ctrl+L: Download current action list
  • Ctrl+S: Save current action list
  • Ctrl+C: Clear the current list
  • Ctrl+K: Execute/Validate the current list
  • Ctrl+F: Action file picker

Toolboxes

  • Ctrl+X: Console
  • Ctrl+P: Attributes Inspector
  • Ctrl+A: Actions Toolbox
  • Ctrl+E: Test Editor Toolbox
  • Ctrl+T: Element Tree Toolbox
  • Ctrl+U: Test Runner Toolbox
  • Ctrl+J: Project Manager
  • Ctrl+Z: Settings Toolbox
  • Ctrl+? (or Ctrl+H): Soda VisualEditor Help
